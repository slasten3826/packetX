# План Исследования X12

## Текущая фаза

Сейчас `X12` находится в исследовательской фазе.

Это значит:

- мы пока не пишем сам display server
- мы пока не проектируем окончательный протокол
- мы пока не делаем преждевременную реализацию

Сейчас задача другая:

- понять, где именно ломаются `X11` и `Wayland`
- проверить, выдерживает ли онтология `X12` реальный системный материал
- выделить примитивы, без которых `X12` не сможет существовать как отдельная система

## Уже сделано

### Прочитано

- верхний контур `slastack`
- `stack-core`
- `ProcessLang`
- `packet`
- исходники `xserver`
- исходники `wayland`

### Уже зафиксировано

- `X11` делает множественность вторичной
- `Wayland` работает хорошо за счет compositor-centric collapse
- `X12` должен проверить, можно ли разжать этот collapse без потери coherence
- выделены первые кандидаты в примитивы `X12`

## Главный исследовательский вопрос

Можно ли построить display/runtime-систему,
в которой:

- множественность первична
- композитор не является центром истины
- relation / form / manifestation разделены сильнее, чем в `Wayland`
- practical coherence не разваливается

## Метод исследования

Мы не начинаем с кода.

Мы начинаем с архитектурных вскрытий.

Для каждого проблемного узла:

1. описать, как он устроен в `Wayland`
2. определить, какие слои там сейчас схлопнуты
3. попытаться разложить его по `chaos / table / crystall / manifest`
4. посмотреть, где этого недостаточно
5. ввести новый примитив `X12`, если он действительно нужен

## Порядок разбора узлов

### Шаг 1: `input routing`

Почему первым:

- это быстрый тест на власть композитора
- тут сразу сталкиваются relation, focus, visibility и form
- здесь быстро видно, достаточно ли `relation` и `focus chain`

Нужно ответить:

- кто в `X12` решает, куда идет input
- зависит ли input routing от final scene
- можно ли отделить focus truth от compositor-owned visibility truth

### Шаг 2: `scene graph ownership`

Почему вторым:

- это один из главных узлов власти в `Wayland`
- здесь решается, можно ли жить без единого compositor-owned world model

Нужно ответить:

- нужен ли вообще один владелец сцены
- чем заменить `scene graph` как центр истины
- достаточно ли для этого `form + visibility graph + manifest path`

### Шаг 3: `output placement`

Почему третьим:

- это напрямую связано с твоей исходной болью: mixed-monitor reality
- здесь вылезает вопрос global space и carrier-модели

Нужно ответить:

- обязателен ли общий desktop-space
- как fields связываются с carriers
- кто определяет shared vs dedicated output behavior

### Шаг 4: `direct scanout / fullscreen path`

Почему четвертым:

- это главный stress-test practical usefulness
- тут проверяется, может ли `manifest` быть реально сильным слоем

Нужно ответить:

- можно ли вынести fullscreen/direct path в first-class manifestation logic
- что должен содержать `scanout contract`
- где проходит граница `manifest <-> chaos`

## Критерий завершения исследовательской фазы

Исследовательская фаза считается достаточно зрелой, когда:

1. разобраны минимум 4 ключевых узла
2. у каждого узла есть документированный вывод
3. список базовых примитивов `X12` стабилизировался хотя бы в версии `v0`
4. становится ясно:
   - `X12` это отдельная система
   - или `X12` это глубокая декомпозиция / переосмысление `Wayland`

## Рабочее правило

Если в какой-то момент появляется желание "поскорее что-нибудь написать",
надо сначала проверить:

- уже ли понятны сущности
- уже ли понятны boundary contracts
- уже ли понятно, какая именно проблема решается

Если нет, значит нужно продолжать исследование.
