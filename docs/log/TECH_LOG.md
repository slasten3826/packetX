# Техлог X12

## Запись 0001: Начальная постановка проблемы

Дата: 2026-04-09

### Мотивация

`X12` начинается с недовольства текущим графическим стеком Linux:

- старый `X11` плохо держит реальность со смешанными мониторами
- `Wayland` лучше работает из коробки, особенно при разном разрешении и разной герцовке
- но `Wayland` выглядит слишком непрозрачным и слишком завязанным на композитор как центр

Цель не в том, чтобы немедленно заменить `Wayland`.
Цель в том, чтобы определить более сильную онтологию display/runtime-системы и проверить ее через постепенную реализацию.

### Первое подтвержденное структурное чтение X11

После чтения `xserver`:

- `X11` слишком рано собирает экранное пространство как нечто общее
- `screenInfo` и `ScreenPtr` являются базовыми мировыми объектами
- многомониторность навешивается сверху через `RandR`
- outputs/CRTC не являются первичной онтологией

Вывод:

`X11` делает множественность вторичной.

### Первое подтвержденное структурное чтение Wayland

После чтения `wayland`:

- `wl_display` является singleton core object
- `wl_registry` публикует global objects
- `wl_compositor` является singleton global
- `wl_output` описан как область в global compositor space
- композитор прямо объявлен display server'ом

Вывод:

`Wayland` решает практические проблемы за счет схлопывания нескольких функций внутрь композитора.

### Рабочая гипотеза X12

`X12` не должен повторять:

- глобально-экранную онтологию `X11`
- онтологию `Wayland`, где композитор становится центром мира

`X12` должен попробовать:

- сделать множественность первичной
- ввести явную модель отношений
- сделать явные translation layers для legacy-систем
- четче разделить:
  - `table`
  - `crystall`
  - `manifest`

### Текущий маппинг слоев

#### Chaos

- Linux kernel
- DRM/KMS
- GPU/VRAM
- input hardware
- vblank/timing
- сырые ограничения substrate

#### Table

- relation server `X12`
- object model
- адресуемость
- protocol truth
- capability routing
- координация topology и lifecycle

#### Crystall

- устойчивая GUI-форма
- логика window manager
- закон раскладки
- устойчивая организация surfaces
- workspace/shell-форма

#### Manifest

- финальная реализация сцены
- функция композитора
- fullscreen-реализация игры
- direct presentation path
- конечный графический объект/событие, уже переживаемое пользователем

### Важное уточнение

Монитор не является `crystall`.

Монитор ближе к:

- `carrier`
- manifestation edge
- физическому или логическому носителю вывода

`Crystall` относится к устойчивой графической форме, а не к железу.

### Вывод о совместимости

Приложениям не нужно понимать внутреннюю философию `X12`.

Им нужны предсказуемые внешние контракты.

Поэтому:

- `X12` может иметь нестандартную внутреннюю онтологию
- но совместимость должна решаться через явные bridges/translators

Кандидаты:

- `x12-x11`
- `x12-wayland`
- позже: native `x12`

### Открытый вопрос

Можно ли декомпозировать `Wayland`, понизив его композитор из центра истины уровня `table` до функции уровня `manifest`?

Возможные исходы:

1. Это работает, и тогда `X12` может вырасти как путь переинтерпретации / форка / декомпозиции.
2. Это не работает, и тогда станет точно видно, где централизация композитора в `Wayland` структурно необходима.

и найти места, где чистая декомпозиция ломается.
