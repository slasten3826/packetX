# X12: Цель И Критерии Успеха

## Зачем этот документ

`X12` уже начинает обрастать сильной онтологией.

Но без жесткой внешней цели проект может скатиться в красивую,
но бесполезную архитектурную мечту.

Поэтому нужно зафиксировать:

- что именно мы вообще делаем
- по какому критерию это будет считаться успехом

## Главная цель

`X12` создаётся не ради абстрактной философии.

Главная цель такая:

> сделать графическую систему для Linux,
> которая работает не хуже `Wayland`,
> но устроена прозрачнее, глубже и сильнее архитектурно

То есть:

- не просто "новая онтология"
- не просто "интересный display experiment"
- не просто "анти-Wayland проект"

А реальная система,
которая в повседневной жизни не хуже того,
что уже есть у пользователя сейчас.

## Короткая формула

`X12` должен быть:

- не хуже `Wayland` practically
- сильнее `Wayland` ontologically

## Что это значит на практике

Если `X12`:

- красиво мыслится
- глубоко устроен
- но работает хуже `Wayland`

то проект не выполнен.

Если `X12`:

- работает не хуже `Wayland`
- и при этом дает новую прозрачную онтологию графического процесса

тогда проект выполнен по-настоящему.

## Базовые практические критерии успеха

### 1. Mixed monitors work

Система должна уверенно держать:

- разные разрешения
- разную герцовку
- разные outputs

без пользовательского ада.

### 2. No architectural tearing

Тиринг не должен быть нормальным состоянием системы.

Если он и возможен из-за vendor/driver chaos,
то не из-за архитектурной неясности самого `X12`.

### 3. Fullscreen and games are strong

- fullscreen path должен быть сильным
- games / Wine / heavy graphics не должны быть второсортным use case

### 4. Visibility-aware work reduction

То, что пользователь не видит,
не должно без нужды жрать полный render/composition budget.

### 5. Practical coherence

Пользователь не должен "бороться со стеком".

Система должна работать цельно:

- outputs
- input
- visibility
- fullscreen
- present

### 6. Compatibility is not pain

Legacy и существующий Linux world
не должны превращать систему в кошмар.

### 7. Driver chaos is survived architecturally

Открытые/закрытые драйверы,
NVIDIA/AMD/прочее —
это часть реальности,
которую `X12` должен выдерживать,
а не игнорировать.

## Внутренняя и внешняя победа

### Внутренняя победа

У `X12` появилась своя настоящая онтология:

- packet
- process
- field
- form
- visibility graph
- manifestation priority

### Внешняя победа

Пользователь ставит систему и говорит:

"оно работает не хуже Wayland"

Только когда есть и то, и другое,
`X12` считается настоящей победой.

## Самый короткий критерий

Если `X12` не работает не хуже `Wayland`,
вся остальная красота не засчитывается.
