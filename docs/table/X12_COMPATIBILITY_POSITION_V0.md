# X12 Compatibility Position v0

## Главный тезис

`X12` строится на своей онтологии,
но не должен терять совместимость с историческим `X`.

Совместимость должна жить на границе,
а не внутри ядра.

## Что это значит

Внутри `X12`:

- своя packet-онтология
- свои слои
- своя серверная truth

Снаружи:

- старые клиенты должны получать мосты совместимости

## Обязательная совместимость

### `X11`

`X11`-совместимость для `X12` должна считаться обязательной.

Причина простая:

- если проект называется `X12`
- и претендует на линию `X`
- то он должен уметь говорить с миром `X11`

Но:

- не через заражение ядра старой онтологией
- а через boundary translator / bridge

Рабочая формула:

> ядро новое, совместимость мостом

## Необязательная совместимость

### `Wayland`

`Wayland`-совместимость на текущем этапе не считается обязательной.

Она может появиться позже,
если будет нужна practically.

Но `X12` не должен строить свое ядро
из обязанности быть `Wayland-compatible`.

## Почему так

Если тащить старую совместимость прямо в ядро,
`X12` перестанет быть новой системой
и снова утонет в чужой онтологии.

Правильнее:

- ядро строится по своим законам
- compatibility живет на boundary

## Что делает X12 все еще X

Минимально:

- server-shaped architecture
- server-side truth
- packet birth only in `chaos`
- client/server boundary
- возможность transport/wire form позже
- обязательный `X11` compatibility bridge

## Короткая формула

`X12` навязывает новую онтологию,
но не режет историческую линию `X`.

`X11` compatibility — must.  
`Wayland` compatibility — optional later.
